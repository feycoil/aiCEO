## Livrables

### 1. Backend — Migration emails JSON → SQLite

**Nouveau fichier** : `data/migrations/2026-04-28-emails.sql`

Table `emails` avec 15 colonnes (id Outlook PK, account, folder, subject, from_name/email, to_addrs, received_at, unread, flagged, has_attach, preview, inferred_project, is_self, **arbitrated_at**, ingested_at) + 6 index (received_at, unread, flagged, inferred_project, arbitrated_at, is_self).

**Patché** : `scripts/normalize-emails.js`
Après production des JSON `data/emails.json` + `data/emails-summary.json`, ingère chaque email dans la table SQLite via `INSERT OR REPLACE` (idempotent sur PK Outlook). Tolérant si table absente.

**Nouveau** : `scripts/ingest-emails.js`
Script standalone de rattrapage. Ingère le `data/emails.json` existant en DB sans relancer Outlook. Utilisé pour migrer les 1052 emails déjà sync'és sans repasser par Outlook COM.

### 2. Backend — Bootstrap auto-création projets + contacts

**Nouveau** : `scripts/bootstrap-from-emails.js`
- 1 projet par valeur distincte de `emails.inferred_project` (skip si name existe déjà)
- 1 contact par expéditeur récurrent (`is_self=0`, ≥3 emails reçus) avec nom le plus fréquent + email + last_seen_at
- Idempotent (peut tourner N fois sans doublons)

**Nouvelle route** : `POST /api/arbitrage/bootstrap-from-emails` (dans `src/routes/arbitrage.js`)
Expose le bootstrap en API REST pour câblage UI futur. Retourne `{projects:{created,skipped,names}, contacts:{created,skipped,names}, totals, message}`.

### 3. Backend — Route arbitrage analyze-emails SQL

**Réécrit** : `POST /api/arbitrage/analyze-emails` dans `src/routes/arbitrage.js`

Avant : cherchait dans table inexistante `events.type='email'` → retournait toujours 0 propositions.

Après : requête SQL directe sur `emails` avec scoring inline :
```sql
score = flagged*100 + unread*30 + has_attach*5
      + (CASE WHEN inferred_project IS NOT NULL THEN 10 ELSE 0 END)
      + (CASE
          WHEN julianday('now') - julianday(received_at) < 1 THEN 20
          WHEN julianday('now') - julianday(received_at) < 3 THEN 10
          WHEN julianday('now') - julianday(received_at) < 7 THEN 5
          ELSE 0
        END)
WHERE is_self = 0 AND arbitrated_at IS NULL
ORDER BY score DESC, received_at DESC LIMIT 8
```

Heuristique `kindFor(subject)` mappe vers `task | decision | project` selon regex.

## Risques résiduels

1. **R1 — LLM 4 surfaces UX restent en preview** (coaching, decision-recommend, auto-draft-review, "si vous tranchez A"). Mitigation : sprint v0.7 dédié.
2. **R2 — Sync events Outlook manquante** — page agenda vide. À implémenter v0.7 (`fetch-outlook-events.ps1` + ingestion table `events`).
3. **R3 — Status decision n'a pas `reportee`** — kanban col "Reporté" volatile (sessionStorage). À ajouter en migration v0.7.
4. **R4 — Scoring `kindFor()` peut classer trop d'items en `task`** — ratio decisions/projects/tasks à monitorer après recette.
5. **R5 — emails n'ont pas de FK vers projects/contacts** — ne ressortent pas dans `/projet/?id=`. À ajouter v0.7.
